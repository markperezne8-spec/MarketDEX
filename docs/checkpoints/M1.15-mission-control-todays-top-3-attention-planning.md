# M1.15 Mission Control Today's Top 3 Attention Planning

**Status:** Planning checkpoint active  
**Canonical branch:** `main`  
**Source of truth:** GitHub repository `markperezne8-spec/MarketDEX`

**Parent checkpoint:** `docs/checkpoints/M1.14H-visual-acceptance-checkpoint-sync.md`

## Milestone intent

M1.15 defines the next safe Mission Control North Star milestone after the completed M1.14 visual shell sequence: a contract-first path for a read-only `Today's Top 3` / attention-priority surface.

The future surface should eventually answer: `What deserves attention first today?`

This planning checkpoint authorizes planning only. It does not implement UI, calculate rankings, create a force-rank engine, display fake live values, execute actions, mutate business data, add automation, or connect to marketplace/network providers.

## Current gap from the North Star

M1.14 moved Mission Control visibly closer to the North Star, but the most important North Star decision area remains contract-first:

- `Today's Top 3` is not yet backed by an approved immutable view model or service contract;
- no approved ranking semantics exist for comparing inventory, listing, pricing, audit, capital, risk, and market attention signals;
- no approved evidence boundary exists for local-only priority items;
- no action execution authority exists from Mission Control priority cards;
- unavailable placeholders must remain honest until the contract exists.

## Safe planning principle

M1.15 treats `Today's Top 3` as an attention contract, not as permission to invent recommendations.

Future implementation may show real priority items only after an immutable view model or service boundary defines:

- the local-only evidence accepted by the surface;
- deterministic item ordering;
- deterministic status states;
- reason text for each item;
- how missing, partial, stale, or error evidence is rendered;
- explicit separation between read-only guidance and any future executable workflow.

## Approved evidence posture

M1.15 future slices may consume only prepared or injected local evidence. Missing evidence must produce `Unavailable` or `Partial` output rather than inferred live status.

Allowed future evidence categories, once explicitly contracted:

- inventory readiness and local inventory summary evidence;
- authority/audit readiness evidence;
- listing workflow readiness evidence;
- local operational readiness evidence already surfaced by approved Mission Control contracts;
- static or prepared local notes that explain why an item is unavailable.

Excluded until separately approved:

- live marketplace prices;
- online trend data;
- automatic sales/listing recommendations;
- polling or background refresh;
- external provider calls;
- mutation commands;
- alerts, notifications, or scheduled tasks.

## Planned deterministic state model

The future contract should preserve these states:

- `Ready` - sufficient local evidence supports showing a read-only priority item;
- `Unavailable` - required evidence is missing or no priority can be safely constructed;
- `Partial` - some local evidence exists, but the item cannot be treated as complete;
- `Error-safe` - injected evidence or contract assembly fails and the UI must render inline safe text.

State labels must be deterministic and plain-language. Error-safe rendering must remain inline and non-modal.

## Planned display contract

The future visible surface should remain compact and read-only. Candidate item fields:

- rank, limited to 1 through 3;
- title;
- status;
- reason;
- evidence summary;
- affected area, such as Inventory, Listing, Authority, Capital, Risk, or Visual Intelligence;
- next safe local preparation note;
- unavailable or partial explanation when evidence is incomplete.

The display must not include action buttons, launch workflows, command execution, popups, alerts, notifications, or automatic remediation.

## Relationship to existing Mission Control surfaces

M1.15 should preserve the accepted top-of-page order unless a future visible build proves a safer adjacent placement:

1. Mission Control header / Command Rail shell;
2. Command Status;
3. System Health;
4. Operational Status;
5. Next Steps;
6. Today's Top 3 / attention-priority surface;
7. Dashboard Grid and lower North Star sections.

The future surface should complement `Next Steps`, not duplicate it:

- `Next Steps` answers: `What is the next safe local step I can prepare for right now?`
- `Today's Top 3` answers: `What are the top three attention items, ranked by approved local evidence?`

## Suggested mergeable build sequence

### M1.15A - Attention priority view-model contract

Create a non-visual immutable view model for the `Today's Top 3` / attention-priority surface. It should accept prepared local evidence through injection only, preserve deterministic rank ordering, and render missing evidence as `Unavailable` or `Partial`.

### M1.15B - First visible Today's Top 3 shell

Place a compact read-only `Today's Top 3` surface in Mission Control using the M1.15A contract. Default missing evidence should render honest unavailable cards. This visible build requires screenshot review.

### M1.15C - Deterministic display states

Lock `Ready`, `Unavailable`, `Partial`, and `Error-safe` visual rendering for priority items, including inline error-safe text and deterministic ordering.

### M1.15D - Visual and contract verification

Record screenshot evidence, placement verification, read-only behavior, injected-only behavior, deterministic ordering, absence of action controls, and absence of popup, polling, networking, persistence, automation, alert, notification, or business mutation paths.

### M1.15E - Checkpoint and documentation sync

Synchronize repository authority after accepted visual evidence and CI have passed.

Each build must remain independently reviewable and mergeable.

## Visual acceptance criteria

Future visible M1.15 builds require screenshot review. Acceptance should confirm:

- the surface is visibly aligned with the North Star `Today's Top 3` concept;
- unavailable evidence is labeled honestly;
- the surface is compact and readable on Mark's desktop viewport;

No runtime behavior, UI implementation, marketplace integration, alerts, notifications, polling, network access, persistence, database migration, task execution, ranking engine, or business mutation is authorized by this planning checkpoint.
