# M1.20 Mission Control Data Freshness Planning

## Purpose

M1.20 defines the next guarded Mission Control contract-first slice after the accepted Visual Intelligence sequence.

Data Freshness is a read-only awareness surface. It may identify the freshness of prepared local evidence, but it does not fetch, refresh, validate, or mutate evidence.

## Authority

The Mission Control command-center surface remains the primary workspace for explaining local operating readiness. Data Freshness must remain subordinate to existing approved Mission Control surfaces and must not replace Capital Health, Opportunity + Risk, Business Scoreboard, or Visual Intelligence.

The future surface may appear as a compact read-only region near the existing readiness surfaces. Exact visible placement requires a separate implementation contract and visual review.

## Evidence boundary

- Evidence must be prepared and injected through an explicit local-only contract.
- The contract may carry source authority, as-of label, freshness label, and freshness state.
- Missing evidence renders Unavailable.
- Incomplete evidence renders Partial and identifies what is missing.
- Valid complete evidence may render Ready.
- Evidence preparation or adapter failure renders Error-safe without crossing the visible boundary.
- No freshness claim may be inferred from process time, current time, file timestamps, or network availability.
- The surface is not an authority and cannot refresh or repair evidence.

## Deterministic behavior

The future contract must lock:

- stable item and region ordering;
- title and subtitle text;
- Ready, Unavailable, Partial, and Error-safe labels;
- source authority and as-of/freshness presentation;
- missing-evidence and failure messages;
- immutable outputs after construction;
- deterministic rejection of invalid inputs and duplicate keys.

## Explicit non-goals

M1.20 planning does not authorize:

- runtime UI changes;
- network access, providers, polling, background workers, schedulers, or startup work;
- persistence, migrations, caching, or database writes;
- alerts, notifications, automation, or task execution;
- formulas, freshness inference, scoring, or invented timestamps;
- action controls, commands, route changes, or business mutation;
- new dependencies or parallel authorities;
- replacement of existing Mission Control surfaces.

## Planned sequence

1. M1.20A: immutable Data Freshness view-model contract.
2. M1.20B: contract-backed read-only shell, if placement is approved.
3. M1.20C: visual and contract verification checkpoint.

Each implementation slice must remain independently reviewable. Any visible implementation requires a desktop screenshot, responsive review when layout changes, and Mark visual acceptance before final merge.

## Next implementation gate

The next build may implement only the immutable Data Freshness view-model contract. It must define the injected evidence shape, deterministic ordering, state matrix, source/as-of labels, and error-safe behavior before any runtime UI work.
