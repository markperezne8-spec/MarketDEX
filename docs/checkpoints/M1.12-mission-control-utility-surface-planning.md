# M1.12 Mission Control Utility Surface Planning

**Status:** Planning active
**Canonical branch:** `main`
**Source of truth:** GitHub repository `markperezne8-spec/MarketDEX`

## Milestone intent

M1.12 defines the next small, safe, visible Mission Control improvement after the completed M1.11 first visible Health status surface.

The approved direction is one compact Mission Control readiness / operational status strip that helps Mark understand what is usable now without changing business state, starting background behavior, or redesigning the dashboard.

This planning checkpoint authorizes planning only. It does not implement UI and does not authorize runtime wiring.

## Smallest safe visible surface

The smallest safe surface is a compact read-only operational status strip on the existing Mission Control surface, placed near the top of Mission Control after the existing header and after the approved `System Health` card unless implementation evidence proves a safer adjacent placement.

The strip should summarize local/offline/business readiness using existing offline/local data only:

- local authority readiness;
- offline-first readiness;
- inventory readiness;
- audit/authority evidence.

The strip should answer a practical Mission Control question:

> What can I trust and use locally right now?

It should act as a scan-first cue, not as a new workflow center. It should point toward existing authority, inventory, and audit evidence already available inside the permanent codebase. It must not duplicate whole operational worksheets or invent new business calculations.

## Approved data and service boundary

Future implementation builds may consume only existing offline/local application evidence through injected read-only view models or already-approved application/service boundaries.

Allowed planning candidates include:

- existing Mission Control snapshot values already used by the root Mission Control surface;
- existing inventory readiness or inventory summary values already computed locally;
- existing audit/authority evidence already preserved by the application;
- existing Health status display evidence only as an adjacent cue, not as a new Health execution path.

Any implementation must preserve composition ownership and remain deterministic, read-only, and side-effect-free. UI code must receive prepared display data through injection instead of opening databases, running Health checks, polling, repairing evidence, or mutating business state.

## Explicit non-goals

M1.12 does not authorize:

- UI implementation in this planning build;
- polling;
- networking;
- startup blocking;
- startup execution;
- database migration;
- persistence changes;
- marketplace integration;
- alerts;
- notifications;
- automated recovery;
- dashboard redesign;
- navigation redesign;
- dependency changes;
- live pricing;
- background workers;
- business-state mutation.

Any future work in those areas requires a separately scoped issue and checkpoint update.

## Mergeable build sequence

### M1.12A - Operational status strip view model

Define the smallest immutable injected view model for the strip. It should preserve deterministic display ordering for local authority, offline-first readiness, inventory readiness, and audit/authority evidence. No UI, startup wiring, polling, persistence, database migration, networking, alerts, notifications, marketplace integration, or business-state mutation.

### M1.12B - First visible strip placement

Place one compact read-only operational status strip on the existing Mission Control surface, preferably near the top after the header and approved Health card. Bind only to the M1.12A view model. No dashboard redesign, navigation redesign, new controls, polling, runtime execution, persistence, or mutation.

### M1.12C - Deterministic display states

Tighten rendering for available, unavailable, partial, and error-safe evidence states. Keep labels plain-language and deterministic. Missing evidence must read as unavailable or partial, not as a failure invented by the UI. No popups, recovery flows, warnings, or background checks.

### M1.12D - Visual and contract verification

Add focused contract and UI placement verification for the operational status strip. Record Mark's visual acceptance evidence and a checklist covering the compact shape, placement, local/offline/inventory/audit wording, unavailable/partial/error-safe states, and absence of action controls.

### M1.12E - Checkpoint and documentation sync

Record the accepted M1.12 visible surface, visual evidence, guardrails, and remaining non-goals in repository authority. This is documentation/checkpoint sync only unless a governance test proves a narrow correction is required.

Each build must remain independently reviewable and mergeable. A later build may not silently absorb polling, startup execution, persistence, marketplace integration, alerts, notifications, redesign, or business-state mutation.

## Visual-check expectations

This planning build requires no visual check because it changes documentation and governance tests only.

Future visible implementation builds require visual review before acceptance. The visual check should confirm:

- the strip is compact and readable on the existing Mission Control surface;
- placement does not displace or redesign the approved `System Health` card unexpectedly;
- local authority, offline-first readiness, inventory readiness, and audit/authority evidence are visible or intentionally unavailable;
- wording clearly communicates read-only operational readiness;
- available, unavailable, partial, and error-safe states are understandable;
- no action controls, popups, alerts, notifications, background activity, navigation change, or business-state mutation occurs.

M1.12D should preserve the accepted screenshot evidence and checklist after Mark's visual review.

## Acceptance gate

M1.12 planning is complete when this checkpoint identifies the smallest safe visible surface, location guidance, approved existing data/service boundary, explicit non-goals, visual-check expectations, and independently mergeable builds.

No runtime behavior, UI implementation, persistence, networking, polling, startup behavior, marketplace integration, alerting, notification, redesign, or business mutation is authorized by this planning checkpoint.
