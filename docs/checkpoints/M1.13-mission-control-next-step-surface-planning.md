# M1.13 Mission Control Next Step Surface Planning

**Status:** Planning checkpoint active
**Canonical branch:** `main`
**Source of truth:** GitHub repository `markperezne8-spec/MarketDEX`

## Milestone intent

M1.13 defines the next small, safe, visible Mission Control improvement after the completed M1.11 `System Health` card and the completed M1.12 `Operational Status` strip.

The approved direction is one compact read-only `Next Steps` surface that communicates action readiness without starting work, executing actions, changing business data, or redesigning Mission Control.

This planning checkpoint authorizes planning only. It does not implement UI and does not authorize runtime wiring.

## Smallest safe visible surface

The smallest safe surface is a compact read-only `Next Steps` surface with an `Action Readiness` contract on the existing Mission Control surface, placed near the top after the existing header, the approved `System Health` card, and the approved `Operational Status` strip unless implementation evidence proves a safer adjacent placement.

The surface should answer a practical Mission Control question:

> What is the next safe local step I can prepare for right now?

The surface should remain a scan-first cue, not a workflow runner, task launcher, or action center. It may summarize only prepared/injected local-only evidence:

- next safe app action category;
- whether inventory data is present enough for next workflow steps;
- whether authority/audit evidence is present enough for protected workflow steps;
- a plain-language readiness note that does not run checks, mutate state, or claim live status.

## Approved input and service boundary

Future implementation builds may consume only prepared local application evidence through an injected read-only view model or already-approved application/service boundary.

The future view-model direction is an immutable injected `NextStepReadinessViewModel` or equivalent private UI-facing display contract. It should preserve:

- a deterministic overall readiness state;
- a concise headline or status label;
- a next safe action category when prepared evidence exists;
- inventory readiness display evidence;
- authority/audit readiness display evidence;
- a plain-language readiness note;
- explicit unavailable and error-safe display text.

UI code must receive prepared display data through injection. UI code must not open databases, query repositories, run Health checks, poll, repair evidence, start workflows, mutate business state, or construct fallback business evidence.

## Deterministic state model

Future visible builds should support deterministic display for:

- `Ready` - prepared evidence supports a next safe local action category and the readiness note can be rendered.
- `Unavailable` - no prepared evidence is available; the surface must say readiness evidence is unavailable rather than running checks.
- `Partial` - some prepared evidence is present while inventory readiness or authority/audit readiness is incomplete.
- `Error-safe` - injected display evidence cannot be rendered cleanly; the surface falls back to inline non-modal error-safe text.

Future rendering should preserve deterministic group order:

1. Next safe action
2. Inventory readiness
3. Authority/audit readiness
4. Readiness note

Missing evidence must render as unavailable or partial. It must not become an invented failure, an automatic workflow trigger, or a live status claim.

## Mergeable build sequence

### M1.13A - Next Steps view model

Define the smallest immutable injected view model for `Next Steps` / `Action Readiness`. It should preserve deterministic state, group order, next action category, inventory readiness, authority/audit readiness, readiness note, unavailable state, partial state, and error-safe state. No UI, startup wiring, polling, persistence, database migration, networking, alerts, notifications, marketplace integration, action execution, or business-state mutation.

### M1.13B - First visible Next Steps placement

Place one compact read-only `Next Steps` surface on the existing Mission Control surface, preferably after the header, approved `System Health` card, and approved `Operational Status` strip, before the KPI cards. Bind only to the M1.13A view model. No dashboard redesign, navigation redesign, action controls, popups, polling, runtime execution, persistence, networking, or mutation.

### M1.13C - Deterministic display states

Tighten rendering for `Ready`, `Unavailable`, `Partial`, and `Error-safe` states. Keep labels plain-language and deterministic. Missing evidence must read as unavailable or partial, not as a failure invented by the UI. No action controls, workflow launchers, warnings, popups, recovery flows, or background checks.

### M1.13D - Visual and contract verification

Add focused contract and UI placement verification for the `Next Steps` surface. Record Mark's visual acceptance evidence and a checklist covering compact shape, placement after existing top Mission Control status surfaces, deterministic group order, prepared/injected-only evidence, read-only behavior, unavailable/partial/error-safe states, and absence of action controls, popups, polling, startup execution, networking, persistence, or business mutation paths.

### M1.13E - Checkpoint and documentation sync

Record the accepted M1.13 visible surface, visual evidence, guardrails, and remaining non-goals in repository authority. This is documentation/checkpoint sync only unless a governance test proves a narrow correction is required.

Each build must remain independently reviewable and mergeable. A later build may not silently absorb action execution, polling, startup execution, persistence, marketplace integration, alerts, notifications, redesign, or business-state mutation.

## Visual-check expectations

This planning build requires no visual check because it changes documentation and governance tests only.

Future visible implementation builds require visual review before acceptance. The visual check should confirm:

- the `Next Steps` surface is compact and readable on the existing Mission Control surface;
- placement does not displace or redesign the approved `System Health` card or `Operational Status` strip unexpectedly;
- the surface communicates readiness, not execution;
- next safe action category, inventory readiness, authority/audit readiness, and readiness note are visible or intentionally unavailable;
- `Ready`, `Unavailable`, `Partial`, and `Error-safe` states are understandable;
- no action controls, buttons, popups, alerts, notifications, background activity, navigation change, or business-state mutation occurs.

M1.13D should preserve the accepted screenshot evidence and checklist after Mark's visual review.

M1.13 planning is complete when this checkpoint identifies the smallest safe visible surface, exact placement guidance, approved prepared/injected evidence boundary, deterministic state model, explicit non-goals, visual-check expectations, and independently mergeable builds.

No runtime behavior, UI implementation, action execution, automation, persistence, networking, polling, startup behavior, marketplace integration, alerting, notification, redesign, public API expansion, or business mutation is authorized by this planning checkpoint.
