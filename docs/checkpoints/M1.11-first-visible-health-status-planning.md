# M1.11 First Visible Health Status Surface

**Status:** Planning active  
**Canonical branch:** `main`  
**Source of truth:** GitHub repository `markperezne8-spec/MarketDEX`

## Milestone intent

M1.11 defines the first deliberately small, read-only visible Health surface after the guarded M1.10 composition-root readiness milestone. It presents approved Health evidence in plain language without changing Health execution, startup, or business state.

## Smallest safe visible surface

The first surface is a compact Health status card or panel added to an existing safe application surface. It contains only:

- a plain-language overall status;
- a short deterministic diagnostic summary or line set;
- an explicit empty/unavailable state when no approved evidence is available;
- an explicit error-safe state when the injected view-model cannot produce display data.

The surface is read-only and receives data through an injected view-model/adapter. It does not own Health checks, runtime composition, scheduling, persistence, or recovery behavior.

## Approved guardrails

- No polling or automatic background checks.
- No startup blocking or implicit startup execution.
- No network access, persistence changes, notifications, or business-state mutation.
- No live monitoring, automated repair, alerts, settings rewrites, migrations, marketplace integrations, inventory changes, portfolio changes, navigation redesign, styling overhaul, or dashboard redesign.
- Existing Health contracts remain the source of status, payload, and diagnostic-line evidence.
- Any visible behavior remains deterministic, offline-first, injected, and side-effect-free.

## Mergeable build sequence

### M1.11A — Read-only Health status view-model

Define the smallest immutable view-model/adapter for UI consumption. It accepts approved Health evidence through injection, preserves explicit status and diagnostic lines, and defines empty/unavailable and error-safe display states. No UI or startup wiring.

### M1.11B — Small Health status card/panel

Place one compact read-only card or panel in an existing safe surface. Bind only to the M1.11A view-model. No navigation redesign, broad refactor, polling, or new execution path.

### M1.11C — Deterministic display states

Add deterministic rendering for available, empty/unavailable, and error-safe states. Preserve plain-language status and stable diagnostic ordering. No alerts, notifications, or automatic recovery.

### M1.11D — Visual and contract verification

Add focused view-model/contract tests and the smallest available UI test coverage. Perform deliberate visual review of the card/panel at representative available, empty, and error-safe states.

### M1.11E — Checkpoint and public documentation synchronization

Record the accepted visible surface, test evidence, visual acceptance, and remaining guardrails in repository authority. Synchronize public documentation only for the approved M1.11 surface.

Each build must remain independently reviewable and mergeable. A later build may not silently absorb startup, polling, persistence, or unrelated UI work.

## Visual-check expectations

This planning build requires no visual check. Visual review begins only with M1.11B and is required before any visible build is considered accepted. Review must confirm:

- the card/panel is compact and readable in the existing safe surface;
- status wording is plain language and matches the injected evidence;
- provider/diagnostic ordering is deterministic;
- available, empty/unavailable, and error-safe states are understandable;
- no unexpected popup, navigation change, background activity, or business-state mutation occurs.

M1.11D records the visual evidence and acceptance decision. If a visual defect is found, the visible build remains incomplete until corrected and rechecked.

## Acceptance gate

M1.11 planning is complete when this checkpoint identifies the smallest safe surface, independently mergeable builds, explicit non-goals, and visual-check expectations. No runtime behavior changes are authorized by this planning document.
