# M1.14 Mission Control Visual North Star Alignment Planning

**Status:** Planning checkpoint active  
**Canonical branch:** `main`  
**Source of truth:** GitHub repository `markperezne8-spec/MarketDEX`  
**Visual source of truth:** `MarketDEX_Mission_Control_Visual_North_Star.png`

## Milestone intent

M1.14 defines the staged path for making the MarketDEX Mission Control home page look much closer to the original Visual North Star while preserving the current GitHub-first workflow, offline-first product boundary, and existing authority protections.

The North Star is a dense command-center dashboard. The current desktop UI has the correct dark foundation and emerging top readiness surfaces, but it is still visually sparse compared with the approved original direction.

This planning checkpoint authorizes planning only. It does not implement UI, add fake live data, create marketplace integrations, introduce polling, or grant new business mutation authority.

## Current gap from the North Star

The current Mission Control surface differs from the North Star in several visible ways:

- the left navigation is simpler and less icon/card driven;
- the header lacks the North Star's high-density brand, mascot, period, date, freshness, health, and app-status band;
- the top dashboard is not yet organized into the North Star's multi-card command-center grid;
- KPI cards are less dense and less color-coded;
- inventory command center, alert, scorecard, and visual intelligence areas are not yet represented with the same hierarchy;
- the current layout is structurally safer but does not yet deliver the original high-energy business OS look.

## Safe alignment principle

M1.14 should treat the North Star as the visual direction, not as permission to invent unsupported live data or new authority.

Visual shells may be introduced only when they are clearly read-only and label unavailable or placeholder evidence honestly. Any real business numbers must come from existing local services or newly approved injected view-model contracts.

## What can be safely matched now

The following can be pursued in early visible builds without new business authority:

- stronger Mission Control visual language using existing design-system tokens;
- denser card styling with blue, red, yellow, green, and white dashboard panels;
- North Star-inspired top header/status band using static or already-known local display data;
- more expressive left navigation treatment if it does not change routing behavior;
- rearranged read-only dashboard sections when existing cards remain intact or are migrated in a controlled way;
- visual-only section containers for future North Star areas when they clearly state evidence is unavailable;
- screenshot-based acceptance checks against the repo North Star image.

## What needs contracts first

The following should not be implemented with invented data. They require dedicated view-model or service contracts before real values appear:

- Today's Top 3 force-ranked actions;
- Capital Health;
- Opportunity + Risk;
- Business Scoreboard;
- Inventory Command Center summary and filters;
- Visual Intelligence;
- Inventory Alerts;
- Data Freshness;
- Workbook Health;
- Force Rank Engine;
- Data Coverage;
- Data Quality;
- Backup Status;
- Knowledge Base.

## Visual-only placeholder boundary

If a build introduces a North Star section before its data contract exists, it must:

- be read-only;
- state that evidence is unavailable or pending a future contract;
- avoid fake scores, fake revenue, fake marketplace values, or fake alerts;
- avoid buttons that execute workflows;
- avoid popups, notifications, background checks, polling, networking, or persistence changes;
- keep business mutation authority out of the UI shell.

## Explicit non-goals

M1.14 does not authorize:

- live marketplace integrations;
- pricing providers;
- network calls;
- polling;
- background workers;
- alerts or notifications;
- task execution;
- action buttons that mutate state;
- fake live dashboard values;
- database migrations;
- persistence changes;
- new dependencies;
- navigation rewrites that break current workspaces;
- business-state mutation;
- broad one-PR redesign of the whole application.

## Suggested mergeable build sequence

### M1.14A - Visual North Star layout map

Create a lightweight layout map that identifies the current Mission Control regions and their North Star target equivalents. No UI implementation. This can include documentation and small tests that protect the visual source-of-truth file path.

### M1.14B - North Star design tokens and panel variants

Extend existing design-system styling for North Star-like panel tones, badges, dense dashboard cards, and section headers. No dashboard rearrangement and no new data authority.

### M1.14C - Header/status band shell

Introduce a compact top status band inspired by the North Star header using static/local/injected evidence only. It may show unavailable status for data freshness, workbook health, or app state when no approved evidence exists.

### M1.14D - Left navigation visual treatment

Update the visual presentation of the existing left navigation toward the North Star style without changing routing, adding workspaces, or altering business behavior.

### M1.14E - Dashboard grid shell

Introduce the North Star-inspired dashboard section layout while preserving existing approved surfaces. Placeholder sections must be read-only and unavailable when evidence is not contracted.

### M1.14F - Inventory command center shell

Add an inventory command center area that groups existing inventory summaries and future placeholders without adding new mutation paths or fake marketplace data.

### M1.14G - Visual intelligence shell

Add lower visual intelligence and alert-region shells with explicit unavailable evidence until contracts are approved.

### M1.14H - Visual acceptance and checkpoint sync

Record screenshot evidence against the North Star image, accepted differences, remaining placeholders, and future contract requirements.

Each build must remain independently reviewable and mergeable. Later builds may not silently absorb networking, polling, live providers, alerts, notifications, persistence, database migration, automation, or business mutation.

## Visual acceptance criteria

Visible implementation builds in M1.14 require screenshot review. Acceptance should compare against `MarketDEX_Mission_Control_Visual_North_Star.png` and confirm:

- the home page is visibly moving toward the North Star command-center style;
- changes preserve offline-first and read-only boundaries;
- unavailable evidence is labeled honestly;
- existing approved Mission Control surfaces are not removed without explicit replacement;
- no fake live values, alerts, marketplace integrations, or action execution are introduced;
- the layout remains usable on Mark's desktop viewport;
- navigation still works as before.

## Acceptance gate

M1.14 planning is complete when the repository records the North Star alignment gap, safe visual-only boundary, contract-first data requirements, explicit non-goals, staged build sequence, and screenshot-based acceptance criteria.

No runtime behavior, marketplace integration, alerts, notifications, polling, network access, persistence, database migration, task execution, or business mutation is authorized by this planning checkpoint.
