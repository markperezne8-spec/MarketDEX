# M1.5 Event Bus Build Manifest

## Status

PLANNING — authority and acceptance boundary

## Objective

Define the smallest safe Event Bus implementation slice before runtime publication or subscription behavior is introduced.

## Ownership

- The Event Bus owns delivery mechanics for approved application events.
- Domain modules remain owners of business facts and event meaning.
- The Event Bus must not become a business database, workflow owner, or hidden mutation authority.
- Composition remains the application wiring authority.
- Delivery remains offline-first, deterministic, and safe at application startup.

## Immutable event envelope

A future runtime event envelope may contain:

- event_name — stable machine-readable event identity;
- occurred_at — event occurrence timestamp;
- source — stable publishing component;
- payload — immutable event-specific values;
- correlation_id — optional operation identifier.

Event names and payload meaning must be owned by the responsibility that publishes the event.

## Boundary rules

The future runtime implementation must:

- deliver only explicitly published events;
- preserve subscriber isolation;
- define deterministic subscriber ordering or explicitly declare ordering unsupported;
- define behavior when one subscriber fails;
- avoid mutating the event after publication;
- avoid implicit network delivery or background automation.

It must not yet:

- replace domain persistence or business history;
- add remote messaging;
- add a UI event monitor;
- create a second composition root;
- silently introduce a global mutable singleton;
- publish undocumented business events.

## Initial implementation boundary

The first runtime build may define immutable event envelopes, pure event validation, and an in-memory publish/subscribe contract.

The first runtime build must not add persistence, remote providers, schedulers, retries, UI surfaces, or startup side effects until separately approved.

## Verification

- focused envelope validation tests;
- subscriber isolation and failure behavior tests;
- startup-safety coverage;
- complete required CI jobs;
- visual check not required for contract-only work.

## Acceptance

M1.5 runtime implementation begins only after this planning authority is merged and the first Event Bus responsibility is explicitly selected.
