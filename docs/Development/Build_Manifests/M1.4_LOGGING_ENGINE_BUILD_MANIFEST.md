# M1.4 Logging Engine Build Manifest

## Status

PLANNING — authority and acceptance boundary

## Objective

Define the smallest safe Logging Engine implementation slice before runtime logging changes are introduced.

## Ownership

- The Logging Engine owns structured diagnostic records, severity classification, formatting boundaries, and diagnostic retention policy.
- Domain modules remain owners of business facts and business-event meaning.
- The Logging Engine must not become a business-event database or an audit-history substitute.
- Composition remains the application wiring authority.
- Logging remains offline-first, deterministic, and usable without network access.

## Structured record contract

A future runtime log record may contain:

- timestamp — locally generated event time;
- level — one of DEBUG, INFO, WARNING, ERROR, or CRITICAL;
- component — stable source component name;
- event — stable machine-readable event name;
- message — human-readable diagnostic context;
- correlation_id — optional identifier for one operation across related records;
- details — optional structured diagnostic values that exclude secrets and credentials.

Records should be deterministic in field naming and safe to inspect as plain text.

## Boundary rules

The future runtime implementation must:

- provide structured records before formatting or persistence;
- keep diagnostic logging separate from business state mutation;
- avoid logging credentials, tokens, payment data, or unnecessary personal information;
- preserve application operation when log storage is unavailable;
- make severity filtering and retention behavior explicit;
- avoid implicit network providers, remote telemetry, or background automation.

It must not yet:

- replace domain history or business-event persistence;
- add remote telemetry or network delivery;
- add a UI log viewer;
- change business calculations or workflow ownership;
- create a second application composition root;
- silently introduce a global mutable singleton.

## Initial implementation boundary

The first runtime build may define immutable log-level and log-record contracts, pure record validation, and deterministic formatting helpers.

The first runtime build must not add file handlers, database tables, migrations, rotation jobs, remote providers, UI surfaces, or startup side effects until those responsibilities are separately approved.

## Verification

- focused contract tests for valid and invalid records;
- deterministic formatting coverage;
- secret-exclusion and unavailable-sink behavior coverage when those contracts are implemented;
- startup-safety coverage;
- complete required CI jobs;
- visual check not required for contract-only work.

## Acceptance

M1.4 runtime implementation begins only after this planning authority is merged and the first Logging Engine responsibility is explicitly selected.
